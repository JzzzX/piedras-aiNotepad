# Piedras 开发手记

## 为什么要做这个

起因是调研 Granola。

国外的会议转写工具很多——Otter.ai、Fireflies、Fathom——但 Granola 走了一条不一样的路：它不追求"全自动帮你记"，而是让你自己写笔记，AI 负责把你的简略记录扩展成完整纪要。录音转写只是底层素材，你写的东西才是核心。

这个思路说白了就是 Human-in-the-loop。不是 AI 替你干活，是 AI 帮你干得更好。

回过头看国内，会议转写这块不缺产品——飞书妙记、腾讯会议纪要、钉钉闪记、通义听悟——但清一色都在卷"全自动"：转写准确率、自动摘要、一键纪要。结果就是生成了一堆没人看的逐字稿和抓不到重点的摘要。

真正在产品评审、用户访谈、销售复盘这些场景里，参会者要的不是万字逐字稿，而是"开会的时候就能抓住关键信息，会后快速出可执行文档"。这个需求在中文世界基本没人好好做。

所以就有了 Piedras——一个面向中文会议场景的"边听边记 + AI 增强"工作台原型。

名字是西班牙语"石头"的复数。会议里的每段对话每个想法都像散落的石子，这个工具帮你把它们拾起来砌成东西。也算致敬 MVP 精神：用最朴素的材料搭最实在的结构。

## 开发过程

技术栈是 Next.js 16 + React 19 + Prisma + Tailwind CSS 4，部署在 Vercel 上。

开发过程中我用 Claude Code Opus 4.6 做整体规划，Codex 5.3 负责主力代码，Gemini 3 处理部分前端工作。我自己主要把控产品决策、代码架构设计和体验细节。

### Day 1：搭骨架

上午从 `create-next-app` 开始，几个小时内核心原型就跑起来了：三栏布局、实时转写、Tiptap 笔记编辑器、AI 对话。

有一个很早就做的决策：Botless 架构。不往会议里塞 Bot，纯浏览器端采集音频——麦克风走 `getUserMedia`，系统音频走 `getDisplayMedia`，双通道混合后送 ASR。前端工程复杂了不少，但用户体验上零侵入。

下午把数据持久化做了（Prisma + SQLite），接了阿里云实时 ASR 替代浏览器 Web Speech API，再接了 Gemini 做 LLM 主通道。LLM 这块做了多 Provider 回退：Gemini → MiniMax → OpenAI-compatible，任何一个挂了自动切下一个。参考国内网络环境下这不是过度设计，是刚需。

晚上到深夜一口气推了 Prompt 参数化、模板系统、Markdown/DOCX 导出、飞书企微 Webhook 分享、跨会议检索问答。Day 1 结束的时候功能已经完整了，但界面还很粗糙。

### Day 2：打磨体验

白天做了品牌切换（设计了 Piedras 的图标和视觉调性，Piedras，西班牙语中"石头"的复数形式，来自我个人比较喜欢的一张indie rock专辑），接了 MCP连接器让外部 Agent能读取会议数据，然后把数据库从本地 SQLite 迁到了云端 PostgreSQL，部署到 `piedras.vercel.app`。

晚上和凌晨基本都在磨移动端适配。反复调筛选面板、模板管理的移动端布局。说实话这块应该 Day 1就用 mobile-first 或者响应式布局来考虑的方式写，拖到后面返工量很大，这也是一个开发途中疏漏的点。

中间有个细节值得一提：尝试了一个比较花哨的浮层筛选方案，最后回退到了简洁的内嵌版本。不只是出于 Demo 的考虑，也是我认为图标设计更直观，less is more.

### Day 3：最后收尾

这天主要在调 AI 助手的空状态——即用户还没输入任何内容时看到的界面。改了六版，从一开始塞太多使用说明，最后精简到一句引导语加几个示例问题。空状态本质上是产品跟用户的第一次对话，塞太多东西反而让人不知道该干嘛, less is more.

最后统一了三栏标题栏样式，补全了 README 和 API 文档。

## 产品设计上的一些思考

### 核心逻辑

Piedras 的产品逻辑可以压缩成一句话：转写提供上下文，笔记承载记录与判断，AI 做结构化放大。

三者有层级关系：转写是客观记录，笔记是用户自己的主观判断和内容扩充，AI 纪要是把判断扩展成完整文档，最后导出、分享或检索。

我觉得这里面最关键的问题是：AI 应该在哪个环节介入？我的答案是不要介入"思考"，而是介入"表达"。你负责想到了什么，AI 负责帮你说完整。

### 跨会议知识库

这是我在 Piedras 里最想做好的一件事，这也是我认为最有潜力和扩展性的一件事。

市面上的会议工具基本都是"单会议"的——录完一场总结一场归档一场，就像一个只能看单张照片不能浏览相册的应用。但真实工作中信息的价值很多时候来自跨会议的关联：

比如在真实音频会议流程中，时常会出现诸如这个需求上次评审时被谁否了？理由是什么？客户上周说的竞品对比，和我们之前的定价讨论矛盾吗？再多轮面试交流中，这个面试应聘者的核心独特优势在哪儿？这些会议流程内容在我之前的一些 SAAS 企业和国企对接实习中深有体会。

这些问题，单会议问答永远答不了。

目前 Piedras 的方案是关键词召回 + 本地哈希向量的混合检索，没有用外部向量数据库，够demo用但不够好。后续要改进的话，应当开新会议时系统自动提醒相关的历史讨论和未解决事项，从"你问它才找"变成"你还没问但你应该知道"。

跨会议知识库还有一个有意思的特性：它的价值是非线性增长的。前几场会议的时候检索没什么用，但积累到足够数量的场次时，你能追溯一个项目从立项到上线的完整决策链条，能发现某某问题已经被提了很多次次但一直没解决。每场新会议不只是多了一条记录，而是让之前所有记录都变得更有价值。

这种复利效应会产生很强的用户粘性——积累了半年会议记录的用户，迁移成本不是搬文件，而是搬记忆，甚至更有可能培养用户的使用习惯。

这种知识库的复利效应在其他公司产品也有体现，我很喜欢的一个例子是一家叫语核科技的startup，他们的创始人在访谈里说到自己的agent数字员工护城河时，其中agent数字员工的培养过程所形成的独特知识库，其他做LLM基模的大厂也无法一时完全攻入。

在中国的工作环境里这个需求可能更强烈。国内企业开会频率高、决策链条长、口头承诺多于书面记录、人员流动又快。一个能帮团队留住"我们为什么这么决定"的系统，解决的不是效率问题，是知识资产流失问题。这个方向上 2B 的付费意愿会远高于单纯的转写。

这个跨会议知识库，更能拓展到“个人知识库”，面向更广阔的场景与生态融合，这点在后文我会阐述。

### Recipe 模版

Piedras 的 AI 纪要不是一个万能 Prompt 直接生成的，而是按会议类型（通用/周会/需求评审/销售沟通/面试复盘）和输出风格（简洁/平衡/详细/行动导向）做了可配置的矩阵。

这个设计也是为了用户的个性化需求，不同用户有不同的产品需求和复杂性，产品应该用场景化模板帮用户消化掉这些复杂性。

## 关于音频笔记市场的一些想法

### 国内赛道

国内 AI 会议赛道看着很热闹，但其实分层很明确：基础设施层（讯飞、阿里云 ASR）只提供 API 不做产品体验；平台捆绑层（飞书妙记、腾讯会议纪要）绑定自家平台、封闭生态；独立工具层（通义听悟）做通用转写加摘要但没有场景深度。

Granola呢，强调人的参与、增强而非替代，这在中国基本是空的。不是没需求，是我认为国内产品的惯性还停在全自动的叙事上（从诸多app追求大而全就可见一斑）。

### 2B 还是 2C

我觉得这个问题本身就问窄了。与其纠结 2B 还是 2C，不如想清楚到底服务谁。

2B-Saas这条路上飞书钉钉已经占位了，独立产品很难打。2B-Workflow（深入行业流程，比如法律庭审、医疗问诊）壁垒高但需要行业积累。2P（Pro-sumer，专业个体）——产品经理、咨询顾问、记者、知识博主——对这些人来说，这不仅是回忆工具，也是思考放大器，付费意愿明确，是最好的切入点。

但如果把视野从"会议工具"扩大到"音频智能"，C 端其实也有很多被低估的场景：

一类是知识创作者。做访谈的博主录完一小时音频要反复回听提取金句，耗时通常是录音时长的三到五倍。播客听众型的创作者每天大量听播客然后提炼整合输出文章，需要的不是转写而是跨多条音频源的知识汇总。在线教育的学习者也需要边听课边记笔记然后 AI 帮整理。这些人的生产力直接取决于处理音频的效率，付费意愿其实很强。

另一类是更日常的场景。很多直播没有字幕——游戏直播、带货直播、海外直播——听障用户或者嘈杂环境下的观众就需要实时转写（我自己在24年底当时看newjeans的直播时就深有感触，gpt-realtime这个多模态模型是一个对创业者来说好的机会）。Piedras 的浏览器端系统音频采集天然支持这个，共享直播标签页就能出字幕。还有线下的讲座、分享会、行业沙龙，大多数人听完就忘，一个打开手机就能录音转写加边听边记的工具能让信息留存率大幅提升。

这些 C 端场景单个体量不大，但直播字幕、讲座课堂笔记比会议记录工具更容易让普通人理解和尝试。可以作为获客入口（毕竟海外直播字幕和课堂笔记，天然有很多热点可以结合推广），后续再往深度的知识管理和专业场景引导。

说到底，2C 的轻量场景可以拉新吸引用户，2P 的专业场景负责用户留存和引导付费变现，2B 的行业可深度扩大营收天花板。底层的产品能力是同一套，面向不同人群做不同包装。

### 中国市场的几个特殊点

中文 ASR 比英文难很多——没有天然词边界，方言、中英混杂、专业术语都是问题。企业协作生态是飞书/钉钉/企微三足鼎立，独立工具必须考虑跟它们的集成（Piedras 第一天就做了飞书和企微的 Webhook 分享）。隐私合规方面采用 Botless 架构，但其实真正面向国内市场的话，音频留存确实也很有必要，我的demo这方面考虑欠佳。这类产品个人用户付费意愿偏低，可能需要"基础免费 + AI 能力付费"或按调用次数计费的模式。

## 与即刻小宇宙生态融合的思考

小宇宙做的是公开的长音频（播客），Piedras 做的是个人的长音频（会议）。底层能力一样：对长音频的理解和结构化。

融合起来有几个直接的场景：

第一，播客笔记。小宇宙有大量学习型听众，如果能边听播客边记笔记、AI 对照播客转写扩展笔记、跨多期播客检索知识，从听完就忘就变成了听完能用。

第二，播客创作者的后期。从几小时录音里提精华、写 show notes、标时间戳——这些后期工作很繁琐，AI 纪要能力可以大幅降低成本，小宇宙也可以切入播客创作者工具的这个付费场景（诸如视频领域经典的抖音与剪映）。

第三，音频内容的语义搜索。小宇宙目前搜索靠标题和描述文本，如果接入跨音频检索，当用户搜索相关内容时，即使没有任何一期播客标题包含这个关键词，也可以搜索到对应的相关内容，这对用户体验上来说可能会有较大提升。

再回到我们的“个人知识库”这个更宏观的想法，会议记录和播客只是"个人音频信息流"的两个来源。一个知识工作者一天里可能上午开会、通勤听播客、下午打客户电话、晚上看行业 talk 回放。如果这些音频来源能统一到一个知识库里做跨源检索和关联，小宇宙就不只是一个内容消费平台，而是变成了知识积累平台。播客从"听过就流走的河水"变成"可以反复查阅的湖泊"。

即刻和小宇宙的核心用户——互联网从业者、创业者、知识工作者——恰好是对这种"音频知识管理"需求最强烈的人群。用户画像天然匹配。

## 回头看

用 Agent 辅助开发最大的感受是：瓶颈不在写代码的速度，在做决策的密度。能十倍速写代码的时候，你需要十倍速做判断——这个功能该不该做？这个交互够不够好？这个方案是不是过度设计了？AI 不会替你说"这个其实没必要"，你得自己当产品经理和质检员。
